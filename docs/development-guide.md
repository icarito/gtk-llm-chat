# Development Guide

## Environment setup (Linux / Arch-family)

System dependencies (PyGObject comes from the system, not pip):

```bash
sudo pacman -S --needed gtk4 libadwaita python-gobject gobject-introspection
```

Create the venv **with system site-packages** so `gi` resolves to the
system PyGObject (avoids compiling pygobject in the venv):

```bash
python -m venv --system-site-packages .venv
.venv/bin/pip install -e .
```

On Debian/Ubuntu the equivalents are `libgtk-4-dev libadwaita-1-dev
python3-gi gobject-introspection`; on other distros see the README.

## Desktop integration (optional, for daily use)

`pip install -e .` gives you the `gtk-llm-chat` command inside the venv,
but nothing puts it in the GNOME launcher or gives it its icon — the
`.desktop` file has to be installed separately. Do that with:

```bash
desktop/install-dev.sh              # symlinks the venv binary into
                                     # ~/.local/bin, installs the
                                     # .desktop and icons under ~/.local
desktop/install-dev.sh --uninstall  # removes all of the above
```

After this, `gtk-llm-chat` works from any terminal (via the
`~/.local/bin` symlink) and the app shows up in the GNOME app grid with
its icon. Re-run it if you recreate the `.venv` (the symlink target
stays the same, so usually you don't even need to).

## Running

```bash
.venv/bin/gtk-llm-chat                 # normal launch (single process, no tray)
.venv/bin/gtk-llm-chat --cid <CID>     # open a specific conversation
.venv/bin/gtk-llm-chat -m <model> -s "system prompt"
DEBUG=1 .venv/bin/gtk-llm-chat         # verbose debug output
.venv/bin/python -m gtk_llm_chat.main --no-llm   # UI-only mode with stubbed llm
```

Quick smoke test without opening a window:

```bash
.venv/bin/python -c "import gi; gi.require_version('Gtk','4.0'); \
gi.require_version('Adw','1'); from gtk_llm_chat import main; print('OK')"
```

API keys and model config are managed by `llm` itself (`llm keys set …`)
and live in `llm.user_dir()` — the app shares them with the CLI.

## Translations

```bash
./update_po.sh      # extract strings and refresh .po catalogs
./compile_po.sh     # build .mo files
./add_language.sh <code>
```

## Packaging

| Target | How |
|---|---|
| Wheel / PyPI | `python -m build` (setuptools-scm versioning from git tags) |
| Frozen app (Win/mac/Linux) | PyInstaller via `build.spec`; CI drives it with `build-ci.py` |
| AppImage | CI workflow `.github/workflows/build-linux.yml` |
| Flatpak | Manifest in `linux/`, uses `linux/shared-modules` submodule |
| Arch | `linux/arch/PKGBUILD` (builds from release tarball) |
| Windows installer | `windows/` + CI `build-windows.yml` |
| macOS bundle | `macos/` + CI `build-macos.yml` |

Release flow: tag `vX.Y.Z` → `.github/workflows/release.yml` builds and
publishes bundles. (Note: a large unfinished rework of `release.yml` sits
in `stash@{0}` — see [branch-inventory.md](branch-inventory.md).)

## Submodules

```bash
git submodule update --init linux/shared-modules
```

- `linux/shared-modules` → flathub shared modules for the Flatpak build.

(The `linux/pystray` submodule was removed in spec 003 along with the
tray applet. Haiku-related pystray experiments still live on
`icarito/pystray@haiku-experiments` if that work resumes.)

## Gotchas

- The venv must be recreated after a system Python major upgrade
  (system-site-packages + PyGObject tie it to the system Python version).
- `gtk_llm_chat/_version.py` is generated by setuptools-scm — never edit.
- E402 is disabled in flake8 because `gi.require_version()` must precede
  `gi.repository` imports.
- Frozen builds on Python 3.13 need the NumPy patch applied in `main.py`
  before any numpy-related import.
